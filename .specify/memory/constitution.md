# BusIsComming Website Constitution

## Core Principles

### I. 产品定位与范围边界

BusIsComming Website 必须作为 BusIsComing Android App 的可信主页，而不是泛化营销页或
通用交通平台。网站服务于想要查询香港巴士路线、管理出行路线的普通市民，核心功能必须
围绕了解软件功能、试用巴士查询功能、下载 App 试用展开；问题反馈和联系开发者是次要
功能。网站不得提供完整出行路线规划，也不得提供地铁、铁路、渡轮、步行或其他非香港巴士
交通工具的路线查询。所有功能描述、截图说明、数据源说明、品牌名称和路线查询能力，必须
能追溯到 Android 主项目的 `AGENTS.md`、`README.md`、产品文档、规格或真实 App 素材。
尚未在 App 中可用的能力，必须明确标注为规划中或未来能力，不能伪装成当前功能。

理由：用户访问主页时需要快速判断这是香港巴士查询 App，并理解它能做什么、不能做什么。

### II. 前后端分离与契约优先

网站必须采用前后端分离架构。前端负责页面渲染、交互、路由、视觉资源和语言切换；
后端负责内容服务、外部数据代理、第三方集成、缓存、错误归一化和安全边界。前后端
之间必须通过明确的 API 或共享契约协作，计划阶段必须记录契约位置、字段含义、错误
格式和版本兼容策略。前端不能依赖后端内部实现，后端不能要求前端硬编码未公开规则。

理由：主页会承载多语言内容和可能的实时交通集成，清晰契约能降低后续演进成本。

### III. 三语国际化与香港语境

所有面向用户的网站文字必须支持繁体中文、简体中文和英文三语切换，语言标识使用
`zh-Hant`、`zh-Hans` 和 `en`。用户可见字符串必须进入 i18n 资源或等价内容系统，不能
散落硬编码在组件中。语言切换入口必须可发现，并持久化用户选择；未选择语言时，网站
必须优先尊重浏览器语言，否则以繁体中文作为香港语境默认语言。路线号、Citybus、
DATA.GOV.HK、ETA、港币价格等官方或领域术语必须保持准确，不得因翻译损害含义。

理由：目标用户覆盖香港本地、内地访客和英文用户，语言体验是核心产品能力。

### IV. 试用查询与可靠降级

试用查询必须限制在香港巴士查询软件的体验范围内，用于帮助用户理解 App 查询能力，而不是
替代完整出行规划工具。任何实时演示、路线数据、ETA、Citybus 或 DATA.GOV.HK 集成都必须
封装在后端或明确的服务边界内，并具备超时、错误处理、缓存策略和可观测日志。外部服务
不可用时，页面必须展示清楚的降级状态，不能编造实时数据。用于回归的 fixture、示例响应
和第三方原文必须保留来源语义；项目自写说明可以翻译，但不能改写外部数据事实。

理由：试用功能要建立信任，错误实时数据和范围外查询都会误导用户。

### V. 可验证交付与每次 Skill 后提交

每个功能必须在 Spec Kit 阶段定义可验证结果：前端改动必须覆盖三语文案、响应式布局、
关键交互和视觉稳定性；后端改动必须覆盖契约、错误状态和外部集成边界。Spec Kit 的
spec、plan、tasks、research、quickstart 等阶段产物必须使用简体中文撰写，代码标识符、
API 名称和第三方原文保持原文。每次执行任何 Spec Kit skill，包括但不限于
`speckit-constitution`、`speckit-specify`、`speckit-plan`、`speckit-tasks` 和
`speckit-implement`，完成产物更新并通过必要验证后，必须自动执行 `git commit`；仅当
提交信息、提交范围或无关工作区改动不清晰时才询问用户。

理由：规范驱动流程只有在产物可读、验证可复现、提交可追踪时才有实际价值。

### VI. 现代界面与可视化评审

网站页面和交互必须采用现代、简洁、优雅的风格。涉及 UI 相关的讨论、方案比较、页面展示
或重要交互说明时，必须优先提供生成图片、截图、设计稿或可视化 mock，让用户直观看到效果；
不得只用文字说明或 ASCII 图来表达 UI 布局。视觉方案必须服务普通市民的高频查询与下载
决策，避免复杂后台感、夸张营销感和无意义装饰。

理由：网站主页的第一印象直接影响下载转化，UI 决策需要被直观看见，而不是只被描述。

### VII. 电脑与手机双端一致可用

网站前端必须同时兼容电脑和手机展示。任何页面、组件、交互流程和内容模块在设计、实现、
评审和验收时，都必须同时考虑桌面端和移动端 viewport；核心信息、语言切换、试用查询、
下载入口、反馈入口和联系入口在两类设备上都必须完整可见、可读、可操作。不得把手机端
或电脑端视为事后补配；如果某项交互在两端需要不同布局，必须保持同等功能覆盖和清晰
状态反馈。

理由：普通市民可能在手机上即时访问，也可能在电脑上了解和下载，任一端不可用都会直接
损害软件主页的基本价值。

### VIII. Figma 驱动的前端规格

任何涉及网站前端页面、组件、布局、视觉状态或交互流程的 spec，在进入后续计划和实现阶段
前，必须产出明确的 Figma UI 设计和交互设计，并在规格中记录 Figma 文件或链接、关键页面
节点、适用 viewport、交互状态和版本说明。后续 plan、tasks 和 implement 阶段必须把该
Figma 文件或链接作为主要视觉与交互参考。Figma 产物必须沉淀在对应 feature 文档中，作为
后续迭代、回归和设计一致性检查的参考资料；若某个功能完全不涉及 UI，必须在规格中写明
N/A 原因。

理由：前端页面的准确和优雅依赖明确设计源，Figma 产物能降低口头描述误差，并为后续迭代
保留可追溯设计依据。

### IX. 服务端 DDD 架构

任何新增或重构的服务端代码必须采用 DDD 目录组织结构和模块边界。后端必须按领域层、
应用层、基础设施层和接口适配层拆分职责：领域实体、值对象、领域服务和领域错误不得依赖
HTTP 框架、数据库、文件系统、第三方 SDK 或前端契约细节；应用层负责编排用例和事务边界；
基础设施层负责文件、数据库、外部 API、缓存、哈希计算等技术适配；接口适配层负责 HTTP、
CLI、定时任务或其他入口的协议转换和错误映射。计划和任务必须列出服务端 bounded context、
核心实体、层级职责、依赖方向和可测试边界；小型功能也不得把领域规则直接塞进 handler、
controller 或路由函数。

理由：网站后端会逐步承接下载、内容服务、外部数据代理和实时查询能力。DDD 边界能保护
领域规则不被框架和基础设施细节污染，降低后续查询、缓存、错误处理和发布管理的演进成本。

### X. OpenAPI 驱动的服务端接口文档

任何新增、修改或移除服务端 HTTP API 的技术方案，在进入实现阶段前，必须同步创建或更新
OpenAPI 3.1 YAML 接口文档。OpenAPI 文档是服务端接口的权威契约来源，必须记录 endpoint、
operationId、请求参数、响应 schema、错误格式、状态码、认证或缓存要求、降级行为和示例。
Swagger UI、Swagger Editor、Redocly 或等价工具可以用于预览、试接口、lint 和 bundle，
但生成页面或代码注释不得替代仓库内的 OpenAPI 源文件。由这些工具生成的 API UI、静态
HTML 文档或可视化接口页面中，项目可控的标题、分组、接口摘要、参数说明、响应说明、错误
说明和示例说明必须使用中文，默认使用简体中文；工具内置且不可配置的控件文案可以保留
工具默认语言。feature 阶段接口文档必须沉淀在对应 `specs/[feature]/contracts/`，实现阶段
必须同步到 `shared/contracts/` 下的长期契约入口；若采用 `openapi/` 子目录或兼容旧路径，
必须在 plan 中记录。后续实现、测试和 agents 理解项目时，必须优先参考该 OpenAPI 文档。

理由：服务端接口是前后端协作和领域知识沉淀的关键边界。OpenAPI-first 能让开发者、工具和
agents 在不阅读 handler 实现的情况下理解接口能力、错误语义和兼容性要求。

### XI. 中文注释与代码可读性

所有新写或重构的代码必须通过清晰命名、聚焦函数和适当注释保持可读性。注释必须用于解释
领域规则、非显而易见的算法、错误映射、外部 API 约束、DDD 边界、状态机、缓存或降级
策略、浏览器兼容处理和重要权衡；不得为显而易见的赋值、简单条件或重复代码字面含义添加
噪音注释。项目自写注释必须使用中文，代码标识符、协议字段、第三方术语、外部 API 原文和
示例值保持原文。修改已有英文注释时，若该注释属于项目自写说明，必须改为中文或补充中文
解释；第三方代码、生成代码和外部协议片段可以保持原文，但不得被当作项目注释规范。

理由：本项目由多个 agents 和阶段产物共同维护，中文注释能降低上下文恢复成本；只为复杂
意图和边界补充注释，才能提升可读性而不制造维护噪音。

### XII. 服务端稳健性与可观测性

服务端代码不得把 `panic` 作为业务控制流、错误返回或分支捷径。领域层、应用层、基础设施层
和接口适配层必须通过领域错误、普通 error、明确返回值或受控响应表达失败；HTTP 入口必须
启用 panic recovery，并把恢复到的异常转换为受控错误响应和错误日志，不能向浏览器泄露堆栈、
密钥、私有路径或内部实现细节。任何由项目代码创建的 goroutine、并发任务、异步回调或定时
任务必须使用统一包装或 `defer recover` 捕获异常，记录任务名、错误、调用上下文和必要堆栈，
并通过 error、channel、context 或监控边界传递失败；不得静默吞掉异常或让子协程 panic 导致
整个进程退出。

服务端必须提供足够日志帮助开发者观测运行状态和定位问题。日志必须覆盖服务启动与关闭、请求
入口、关键业务结果、外部 API 或文件系统调用、缓存命中或失效、降级状态、错误映射、后台任务
开始与结束、后台任务失败和 panic recovery。日志字段必须包含足够定位问题的上下文，例如
请求路径、operationId、bounded context、任务名、错误类型、状态码或降级原因；不得输出密钥、
token、完整私有路径、用户个人敏感信息或未经脱敏的第三方响应。新增服务端路径必须配套测试或
验证步骤，确认 panic recovery、协程 recover 和关键日志可用；没有自建并发逻辑时必须在 plan
或 tasks 中写明 N/A。

理由：主页后端承担下载、内容服务和未来查询代理能力。单个 handler、外部依赖或子协程异常
不得拖垮整个服务；结构化且脱敏的日志能让开发者快速定位下载失败、降级和运行时异常。

## 技术与内容约束

- 项目结构必须显式区分前端、后端和共享契约；若某个阶段暂未创建完整目录，也必须在
  plan 中说明演进路径和临时边界。
- 首屏必须让用户立即识别产品、核心用途和香港巴士查询场景，优先使用真实 App 截图、
  真实能力说明或可信素材，不能只展示抽象宣传文案。
- 主要信息架构必须优先覆盖：软件功能介绍、试用巴士查询、下载 App、问题反馈、联系开发者。
- 任何路线查询能力都必须保持在香港巴士查询范围内，不得扩展为跨交通工具路线规划。
- 前端必须适配移动端和桌面端，语言切换、主要行动按钮、产品截图和核心说明不得在常见
  viewport 下重叠、截断或不可操作。
- UI 布局、交互和内容展示必须同时设计手机与电脑形态；关键流程至少覆盖一个手机 viewport
  和一个电脑 viewport 的验证证据。
- UI 方案讨论和展示必须提供图片、截图、设计稿或可视化 mock；ASCII 布局图只能作为内部
  结构草稿，不能作为给用户确认 UI 的主要表达。
- 涉及前端 UI 的 spec 必须记录 Figma 文件或链接，并说明设计覆盖的页面、组件、交互状态、
  手机 viewport 和电脑 viewport。
- Figma 文件或链接必须沉淀在对应 feature 的文档中；后续阶段不得只凭文字描述实现页面。
- 三语内容必须保持同等功能覆盖；新增页面、组件、按钮、状态和错误提示时，必须同时补齐
  `zh-Hant`、`zh-Hans` 和 `en`。
- 后端不得向浏览器泄露密钥、私有 token、未公开第三方参数或可绕过服务边界的内部地址。
- 服务端代码目录必须体现 DDD 分层；`domain` 只能依赖标准语言能力和领域自身概念，
  `application` 只能编排领域规则与端口，`infrastructure` 和 `interfaces` 通过端口适配领域。
- 新写或重构的项目代码必须在复杂领域规则、非显而易见算法、错误映射、外部约束、状态转换、
  缓存和降级策略处补充中文注释；不得用注释重复代码字面含义。
- 服务端入口必须启用请求日志和 panic recovery；新增 goroutine、异步回调或定时任务必须通过
  recover 包装保护，并把失败记录为可观测日志或传递到调用方。
- 服务端日志必须覆盖启动、请求、错误映射、外部依赖、缓存、降级和后台任务状态，并对密钥、
  token、私有路径、用户个人敏感信息和第三方原始响应做脱敏或避免输出。
- 涉及服务端 HTTP API 的变更必须维护 OpenAPI 3.1 YAML 文档；文档必须作为接口契约源文件
  纳入 feature contracts 和共享契约，不能只存在于代码注释、截图、Swagger UI 页面或外部服务。
- OpenAPI 文档必须可被 lint 或等价检查验证，并能被 Swagger UI、Redocly 或等价工具预览；
  实现阶段不得在 OpenAPI 文档缺失或明显过期时修改服务端接口。
- 生成的 API UI 或静态接口文档必须以中文呈现项目可控内容；若工具无法本地化内置控件，
  必须在 plan 或 quickstart 中说明限制，并确保接口标题、摘要、描述和示例仍为中文。
- 外部 API 样例、fixture、官方术语和第三方文本必须保留原始语义；为了文风统一而改写
  来源内容属于违规。

## 开发流程与质量门禁

- `/speckit-specify` 产物必须用简体中文描述用户旅程、需求、成功标准和假设。
- `/speckit-specify` 涉及前端 UI 时，必须产出并记录 Figma 文件或链接、关键节点和交互状态。
- `/speckit-specify` 涉及服务端 HTTP API 时，必须说明 OpenAPI 文档的创建或更新要求、长期
  沉淀位置和验证方式。
- `/speckit-plan` 必须在宪法检查中逐项记录来源追溯、前后端边界、三语策略、
  范围排除、Figma 设计引用、UI 可视化、电脑与手机双端适配、OpenAPI 接口文档、外部集成
  降级、API UI 中文输出、中文注释与代码可读性策略、服务端 panic/recover 与日志策略、
  验证计划和服务端 DDD 边界、skill 后提交策略。
- `/speckit-tasks` 必须把任务按用户故事组织，并包含必要的 i18n、契约、视觉图片或截图、
  Figma 文件或链接沉淀、OpenAPI 文档更新、中文 API UI 生成和 lint/预览验证、电脑与手机
  viewport 验证、范围排除、服务端 DDD 分层任务、中文注释与代码可读性任务、服务端
  panic recovery、协程 recover 和日志验证任务、集成验证任务。
- `/speckit-implement` 完成后必须运行与变更范围匹配的验证；涉及前端体验时，必须启动本地
  服务并用浏览器验证关键页面；涉及后端契约或服务端实现时，必须运行对应测试或等价检查，
  确认 OpenAPI 文档与实现一致、生成的 API UI 项目可控内容为中文，并确认领域层没有依赖
  框架、文件系统、数据库或前端契约；涉及新写或重构代码时，必须确认复杂逻辑、领域规则、
  错误映射、外部约束和降级策略已有适当中文注释，且没有重复代码字面含义的噪音注释；涉及
  服务端代码时，必须确认没有以 panic 表达业务错误，HTTP 入口已启用 recovery 和请求日志，
  自建 goroutine 或后台任务已加 recover 包装，并具备足够定位问题的脱敏日志。
- 任何 Spec Kit skill 验证通过后必须提交本次改动。提交信息使用简洁 Conventional Commit
  风格；若工作区存在明显无关改动，必须只提交本次范围或先向用户确认。
- 任何违反宪法门禁的计划或任务必须在复杂度跟踪或等价章节中记录原因、风险和
  被拒绝的更简单方案。

## Governance

本宪法优先于本仓库内其他流程说明。Android 主项目文档仍是 App 事实来源；当网站内容与
主项目事实冲突时，必须先更新事实来源或调整网站描述。

修订必须通过 `speckit-constitution` 执行，并同步检查 plan、spec、tasks、checklist 模板和
运行时 agent 指南。每次修订必须在文件顶部保留 Sync Impact Report，说明版本变化、原则变化、
同步文件和遗留事项。

版本遵循语义化规则：移除或重新定义核心原则为 MAJOR；新增原则、章节或实质扩展治理要求为
MINOR；文字澄清、错别字和非语义调整为 PATCH。1.0.0 为首次制定；1.1.0 新增 UI 和
Spec Kit 提交治理要求；1.2.0 新增双端一致可用核心原则；1.3.0 新增 Figma 驱动的前端规格
核心原则；1.4.0 新增服务端 DDD 架构核心原则；1.5.0 新增 OpenAPI 驱动的服务端接口文档
核心原则；1.5.1 明确生成 API UI 的项目可控内容必须使用中文；1.6.0 新增中文注释与代码
可读性核心原则；1.7.0 新增服务端稳健性与可观测性核心原则。

合规检查必须在计划阶段进入第 0 阶段前完成，并在第 1 阶段设计后复查。代码实现完成前，任务
清单必须能映射到本宪法的可验证门禁；未验证的交付不能标记完成。
