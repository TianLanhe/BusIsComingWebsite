# BusIsComming Website Constitution

## Core Principles

### I. 产品定位与范围边界

BusIsComming Website 必须作为 BusIsComing Android App 的可信主页，而不是泛化营销页或
通用交通平台。网站服务于想要查询香港巴士路线、管理出行路线的普通市民，核心功能必须
围绕了解软件功能、试用巴士查询功能、下载 App 试用展开；问题反馈和联系开发者是次要
功能。网站不得提供完整出行路线规划，也不得提供地铁、铁路、渡轮、步行或其他非香港巴士
交通工具的路线查询。所有功能描述、截图说明、数据源说明、品牌名称和路线查询能力，必须
能追溯到 Android 主项目的 `AGENTS.md`、`README.md`、产品文档、规格或真实 App 素材。
尚未在 App 中可用的能力，必须明确标注为规划中或未来能力，不能伪装成当前功能。

理由：用户访问主页时需要快速判断这是香港巴士查询 App，并理解它能做什么、不能做什么。

### II. 前后端分离与契约优先

网站必须采用前后端分离架构。前端负责页面渲染、交互、路由、视觉资源和语言切换；
后端负责内容服务、外部数据代理、第三方集成、缓存、错误归一化和安全边界。前后端
之间必须通过明确的 API 或共享契约协作，计划阶段必须记录契约位置、字段含义、错误
格式和版本兼容策略。前端不能依赖后端内部实现，后端不能要求前端硬编码未公开规则。

理由：主页会承载多语言内容和可能的实时交通集成，清晰契约能降低后续演进成本。

### III. 三语国际化与香港语境

所有面向用户的网站文字必须支持繁体中文、简体中文和英文三语切换，语言标识使用
`zh-Hant`、`zh-Hans` 和 `en`。用户可见字符串必须进入 i18n 资源或等价内容系统，不能
散落硬编码在组件中。语言切换入口必须可发现，并持久化用户选择；未选择语言时，网站
必须优先尊重浏览器语言，否则以繁体中文作为香港语境默认语言。路线号、Citybus、
DATA.GOV.HK、ETA、港币价格等官方或领域术语必须保持准确，不得因翻译损害含义。

理由：目标用户覆盖香港本地、内地访客和英文用户，语言体验是核心产品能力。

### IV. 试用查询与可靠降级

试用查询必须限制在香港巴士查询软件的体验范围内，用于帮助用户理解 App 查询能力，而不是
替代完整出行规划工具。任何实时演示、路线数据、ETA、Citybus 或 DATA.GOV.HK 集成都必须
封装在后端或明确的服务边界内，并具备超时、错误处理、缓存策略和可观测日志。外部服务
不可用时，页面必须展示清楚的降级状态，不能编造实时数据。用于回归的 fixture、示例响应
和第三方原文必须保留来源语义；项目自写说明可以翻译，但不能改写外部数据事实。

理由：试用功能要建立信任，错误实时数据和范围外查询都会误导用户。

### V. 可验证交付与每次 Skill 后提交

每个功能必须在 Spec Kit 阶段定义可验证结果：前端改动必须覆盖三语文案、响应式布局、
关键交互和视觉稳定性；后端改动必须覆盖契约、错误状态和外部集成边界。Spec Kit 的
spec、plan、tasks、research、quickstart 等阶段产物必须使用简体中文撰写，代码标识符、
API 名称和第三方原文保持原文。每次执行任何 Spec Kit skill，包括但不限于
`speckit-constitution`、`speckit-specify`、`speckit-plan`、`speckit-tasks` 和
`speckit-implement`，完成产物更新并通过必要验证后，必须自动执行 `git commit`；仅当
提交信息、提交范围或无关工作区改动不清晰时才询问用户。

理由：规范驱动流程只有在产物可读、验证可复现、提交可追踪时才有实际价值。

### VI. 现代界面与可视化评审

网站页面和交互必须采用现代、简洁、优雅的风格。涉及 UI 相关的讨论、方案比较、页面展示
或重要交互说明时，必须优先提供生成图片、截图、设计稿或可视化 mock，让用户直观看到效果；
不得只用文字说明或 ASCII 图来表达 UI 布局。视觉方案必须服务普通市民的高频查询与下载
决策，避免复杂后台感、夸张营销感和无意义装饰。

理由：网站主页的第一印象直接影响下载转化，UI 决策需要被直观看见，而不是只被描述。

## 技术与内容约束

- 项目结构必须显式区分前端、后端和共享契约；若某个阶段暂未创建完整目录，也必须在
  plan 中说明演进路径和临时边界。
- 首屏必须让用户立即识别产品、核心用途和香港巴士查询场景，优先使用真实 App 截图、
  真实能力说明或可信素材，不能只展示抽象宣传文案。
- 主要信息架构必须优先覆盖：软件功能介绍、试用巴士查询、下载 App、问题反馈、联系开发者。
- 任何路线查询能力都必须保持在香港巴士查询范围内，不得扩展为跨交通工具路线规划。
- 前端必须适配移动端和桌面端，语言切换、主要行动按钮、产品截图和核心说明不得在常见
  viewport 下重叠、截断或不可操作。
- UI 方案讨论和展示必须提供图片、截图、设计稿或可视化 mock；ASCII 布局图只能作为内部
  结构草稿，不能作为给用户确认 UI 的主要表达。
- 三语内容必须保持同等功能覆盖；新增页面、组件、按钮、状态和错误提示时，必须同时补齐
  `zh-Hant`、`zh-Hans` 和 `en`。
- 后端不得向浏览器泄露密钥、私有 token、未公开第三方参数或可绕过服务边界的内部地址。
- 外部 API 样例、fixture、官方术语和第三方文本必须保留原始语义；为了文风统一而改写
  来源内容属于违规。

## 开发流程与质量门禁

- `/speckit-specify` 产物必须用简体中文描述用户旅程、需求、成功标准和假设。
- `/speckit-plan` 必须在宪法检查中逐项记录来源追溯、前后端边界、三语策略、
  范围排除、UI 可视化、外部集成降级、验证计划和 skill 后提交策略。
- `/speckit-tasks` 必须把任务按用户故事组织，并包含必要的 i18n、契约、视觉图片或截图、
  范围排除、集成验证任务。
- `/speckit-implement` 完成后必须运行与变更范围匹配的验证；涉及前端体验时，必须启动本地
  服务并用浏览器验证关键页面；涉及后端契约时，必须运行对应测试或等价检查。
- 任何 Spec Kit skill 验证通过后必须提交本次改动。提交信息使用简洁 Conventional Commit
  风格；若工作区存在明显无关改动，必须只提交本次范围或先向用户确认。
- 任何违反宪法门禁的计划或任务必须在复杂度跟踪或等价章节中记录原因、风险和
  被拒绝的更简单方案。

## Governance

本宪法优先于本仓库内其他流程说明。Android 主项目文档仍是 App 事实来源；当网站内容与
主项目事实冲突时，必须先更新事实来源或调整网站描述。

修订必须通过 `speckit-constitution` 执行，并同步检查 plan、spec、tasks、checklist 模板和
运行时 agent 指南。每次修订必须在文件顶部保留 Sync Impact Report，说明版本变化、原则变化、
同步文件和遗留事项。

版本遵循语义化规则：移除或重新定义核心原则为 MAJOR；新增原则、章节或实质扩展治理要求为
MINOR；文字澄清、错别字和非语义调整为 PATCH。1.0.0 为首次制定；本次新增 UI 和 Spec Kit
提交治理要求，因此升级为 1.1.0。

合规检查必须在计划阶段进入第 0 阶段前完成，并在第 1 阶段设计后复查。代码实现完成前，任务
清单必须能映射到本宪法的可验证门禁；未验证的交付不能标记完成。
