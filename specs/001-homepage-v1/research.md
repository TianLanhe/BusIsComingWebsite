# 第 0 阶段研究：第一版网站主页

## 决策 1：前端采用 React + Vite + TypeScript

**Decision**：在 `frontend/` 下创建 React + Vite + TypeScript 应用，作为第一版静态主页实现基础。

**Rationale**：仓库当前没有既有前端框架。首页包含轮播、下载按钮交互、三语切换、FAQ 展开和响应式布局，使用 React 可以把页面区块、交互状态和 i18n 内容拆成清晰组件；Vite 适合静态主页快速开发、构建和本地预览。

**Alternatives considered**：
- 纯 HTML/CSS/JS：依赖更少，但三语内容、组件状态和测试维护会更散。
- Next.js：能力完整，但第一版没有服务端渲染或多页面路由需求，复杂度偏高。
- Vue/Svelte：同样可行，但本仓库没有既有约束，React + Vite 是更保守的默认选择。

## 决策 2：第一版不启用实时后端，使用共享契约保留边界

**Decision**：第一版只实现静态前端和共享契约；在线查询为静态演示。后端目录仅记录未来服务边界，不提供实时查询 API。

**Rationale**：规格明确第一版在线查询不是实时能力。过早接入 Citybus 或 DATA.GOV.HK 会扩大范围，也会触发缓存、超时、错误归一化和可观测性要求。通过 `contracts/` 定义首页内容、下载 manifest 和 UI 状态，可以满足前后端分离的计划门禁，并为后续真实在线查询 feature 留出接口边界。

**Alternatives considered**：
- 立即实现后端查询代理：不符合第一版静态演示范围，风险和验证成本过高。
- 完全不定义后端边界：会违反宪法的契约优先要求。

## 决策 3：三语内容使用结构化内容模型

**Decision**：所有用户可见文字进入结构化内容模型，每个可见字符串使用 `zh-Hant`、`zh-Hans`、`en` 三个键。

**Rationale**：规格要求三语同等覆盖；结构化内容模型能让页面区块、FAQ、轮播、下载状态和在线查询限制统一验证，避免组件内散落硬编码。

**Alternatives considered**：
- 组件内硬编码三语：短期快，但难以验证完整性。
- 仅运行时翻译表：可行，但首页内容本身有结构，使用内容模型更利于后续内容服务接入。

## 决策 4：下载按钮同时支持 hover、focus 和触控等价状态

**Decision**：下载按钮状态模型包含 `default`、`android-expanded`、`iphone-expanded`；桌面 hover、键盘 focus 和触控点击都映射到相同语义状态。Android 可进入可下载状态，iPhone 只能进入不可用说明状态。

**Rationale**：用户确认的交互以 hover 为主，但宪法要求手机端同等可用。把 hover 视觉转换成状态机可以让鼠标、键盘和触控共享行为，也能明确 iPhone 不跳转。

**Alternatives considered**：
- 只做 CSS hover：手机和键盘用户体验不完整。
- 拆成两个独立按钮：与用户确认的“一整个分段按钮”不一致。

## 决策 5：首屏轮播使用真实截图优先、临时 mock 显式标注

**Decision**：轮播项需要绑定截图素材来源和状态；真实 Android App 截图为准，临时高保真 mock 只能作为待替换素材。

**Rationale**：网站必须作为可信 App 主页，核心能力说明要追溯到 Android 主项目事实。素材状态写入模型和任务后，后续实现能明确哪些图片不能作为最终交付。

**Alternatives considered**：
- 全部使用 mock：推进快，但可信度弱，且与宪法“真实 App 截图优先”冲突。
- 只用文字说明：不符合 UI 可视化评审要求。

## 决策 6：视觉实现以 Figma 为主源，浏览器截图做验收对照

**Decision**：实现阶段必须使用 Figma 文件和关键节点作为视觉主源，并通过桌面/手机浏览器截图验证首屏、下载按钮状态、轮播状态和在线查询演示。

**Rationale**：宪法要求 Figma 驱动；用户也强调 UI 相关讨论必须用图片。浏览器截图验证能发现文字溢出、响应式布局和交互状态问题。

**Alternatives considered**：
- 只按文字 spec 实现：不满足项目门禁。
- 只看 Figma 不跑浏览器：无法验证真实布局、字体渲染和交互状态。

## 决策 7：测试策略覆盖内容完整性、状态行为和视觉稳定

**Decision**：实现阶段至少包含内容/i18n 完整性检查、组件状态测试、构建检查和浏览器端桌面/手机验证。

**Rationale**：第一版功能虽然是静态主页，但高风险点在三语缺失、按钮状态误跳转、iPhone 状态误导、移动端溢出和在线查询范围误导。测试应围绕这些风险，而不是只验证构建成功。

**Alternatives considered**：
- 只做手动浏览：容易漏掉三语和状态回归。
- 只做单元测试：无法覆盖 Figma 视觉和双端布局。
