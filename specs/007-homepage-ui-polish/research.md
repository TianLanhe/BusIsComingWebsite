# 研究：首页 UI 体验优化补充

## 决策 1：桌面 hero 采用中等放大而不是极限放大

**Decision**：桌面 hero 截图组采用中等放大，主截图目标约为当前版本的 1.25 倍，后排堆叠图跟随同等比例放大；不改变桌面功能介绍区布局。

**Rationale**：用户指出桌面右侧展示区红框内截图偏小。中等放大能提升真实 App 截图的视觉权重，同时保留功能标题和说明的可读宽度，避免 hero 从“功能展示”变成“截图海报”。

**Alternatives considered**：

- 保持现状：改动最小，但无法解决用户指出的截图偏小。
- 极限放大：截图更抢眼，但会挤压文字区，英文和繁中文字容易异常换行。

## 决策 2：大图查看使用轻量 modal，缩放和平移在本地状态中处理

**Decision**：点击主截图打开大图查看模式；modal 支持缩放、平移、关闭和同功能截图切换。实现优先使用现有 React 状态、Pointer/Wheel/Keyboard 事件和 CSS transform，不新增第三方 pan/zoom 依赖。

**Rationale**：需求聚焦于查看 App 截图细节，不需要复杂编辑器级缩放能力。原生实现足以覆盖点击放大、缩放、平移、Esc 关闭、焦点限制和同组截图切换，且避免增加首屏体积和维护依赖。

**Alternatives considered**：

- 引入第三方缩放库：功能丰富，但增加依赖、样式和无障碍集成成本。
- 只打开原图新标签：实现简单，但用户离开当前页面，无法控制同功能截图切换和焦点体验。

## 决策 3：截图区和文字区使用分离手势目标

**Decision**：截图区拖动/滑动只切换同功能截图；文字介绍区拖动/滑动切换整个功能；场景圆点继续切功能；后排堆叠图点击只切当前功能主图。

**Rationale**：用户明确要求“当滑动/拖动折叠图片时，仅对图片进行左右切换；当滑动/拖动文字介绍时，对整个功能进行切换”。这要求事件处理不能继续绑定为整个 carousel 单一拖动目标。

**Alternatives considered**：

- 保持整个 carousel 拖动切功能：与用户要求冲突。
- 只保留点击，不支持拖动：降低误触，但无法满足截图区和文字区分别滑动的交互要求。

## 决策 4：手机功能介绍采用 2 列紧凑卡片

**Decision**：手机端功能介绍采用 2 列紧凑卡片，不加分组胶囊；每张卡保留图标、标题和一句短说明，降低 padding、gap 和单卡高度；桌面端保持现状不变。

**Rationale**：2 列能明显缩短 6 个功能的展示高度，并能扩展到 10 个主要功能；相比 3 列，2 列给繁体中文和英文标题更多空间，避免变成图标墙。

**Alternatives considered**：

- 单列压缩列表：改动最小，但未来 10 个功能仍然过长。
- 3 列卡片：高度更短，但标题和说明空间不足，三语下容易拥挤。
- 横向 carousel：页面最短，但隐藏功能，用户需要滑动才知道还有哪些主要能力。

## 决策 5：手机路线结果卡使用“标签 + 值”的单行指标带

**Decision**：手机路线结果卡保留三层：路线号/候车状态、站点路径或缺失提示、车费/耗时/步行单行指标带。每个指标显示项目名称和值，标签更浅更小，值更深更重。

**Rationale**：用户指出只把 `$4.90 · 9 分钟 · 步行 269 米` 合并为一行不够清楚，必须展示“车费”“耗时”“步行”等项目名，并区分名称和值。该方案比三块纵向指标更省高度，也比纯文本摘要更可扫读。

**Alternatives considered**：

- 三列指标块：清晰但高度偏高。
- 纯文本摘要：省高度但标签和值不够清楚。
- 保持现状单列：信息断裂，首屏只能看到很少路线。

## 决策 6：费用文案使用产品价值表达

**Decision**：功能标题改为 `車費一眼看清 / 车费一眼看清 / Fare at a glance`，说明改为同页比较车费、行程时间和步行距离以辅助选择路线。

**Rationale**：用户明确指出“比较城巴方案时，可直接看到多程全程总车费，而不只是币种显示。”是修改要求，不应作为页面文案。页面应表达用户价值，而不是内部实现意图。

**Alternatives considered**：

- 保留“多程总车费一眼看清”：过窄且用户要求改成“车费一眼看清”。
- 描述“不是币种显示”：属于内部问题说明，不适合作为用户可见文案。

## 决策 7：Figma 采用本地插件 fallback

**Decision**：目标 Figma 文件沿用现有 `BusIsComing Website - Homepage v1 Spec`。计划阶段因 Figma MCP 返回重新认证要求，生成本地 Figma 插件作为可复现设计源，并在 `figma.md` 记录计划节点和回填要求。`2026-06-30` 用户提供真实节点报告后，已将 `Homepage UI Polish - 007` 页面下的 6 个关键 Node ID 回填到 `figma.md` 和相关契约。

**Rationale**：宪法要求涉及 UI 的 spec/plan 有 Figma 设计引用；计划阶段不能直接写入 Figma 时，不能伪造节点 ID。本地插件能在认证恢复或人工打开 Figma 后生成一致节点；真实节点回填后，插件保留为节点缺失或重建时的 fallback。

**Alternatives considered**：

- 只保留 HTML 原型：能表达视觉，但不满足 Figma 产物沉淀要求。
- 伪造节点 ID：不可追溯，后续实现会引用不存在的设计节点。
