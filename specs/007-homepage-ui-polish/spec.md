**功能分支**：`codex/007-homepage-ui-polish`

**创建日期**：2026-06-27

**状态**：Draft

**输入**：用户描述："对网站界面继续优化：桌面首页右侧功能展示区和截图需要适当放大；示例图片支持点击放大并缩放查看细节；截图区与文字介绍区的滑动目标要分离；手机端功能介绍太长，需要改成更紧凑且可扩展到 10 个功能的展示；“多程总车费一眼看清”改为“车费一眼看清”并重写配套介绍；路线查询结果卡片在手机端需要优化。经视觉讨论确认：手机功能介绍采用 2 列紧凑卡片，不加分组胶囊，桌面端功能介绍保持现状；路线卡片采用标签和值同一行但样式区分；桌面 hero 采用中等放大，后排堆叠图同等放大，不显示放大提示器；大图模式左右滑动只切换同功能截图。"

## 用户场景与测试（必填）

### 用户故事 1 - 在桌面首屏更清楚地查看 App 截图（优先级：P1）

桌面用户进入首页后，希望右侧功能展示区的 App 截图足够醒目，能看出真实 App 界面，而不是在大片留白中显得偏小。用户还希望点击主截图后进入大图查看模式，放大细节时不会被切到另一个功能场景。

**优先级原因**：首屏截图是用户判断 App 真实感和精致度的关键区域；当前截图偏小会削弱主页可信度。

**独立测试**：在 1440px 桌面视口打开首页，查看 hero 右侧功能展示区；点击主截图进入大图模式，执行放大、平移、同功能截图切换和关闭。

**验收场景**：

1. **给定** 用户在桌面端打开首页，**当** 查看首屏右侧功能展示区，**则** 主截图比当前版本更醒目，后排堆叠截图也按同等比例放大，且说明文字仍清晰可读。
2. **给定** 用户看到首屏主截图，**当** 点击主截图，**则** 页面打开大图查看模式，并保持在当前功能截图组。
3. **给定** 用户正在大图模式查看截图，**当** 左右滑动或使用切换控件，**则** 只切换同一功能内的截图，不跨功能场景。
4. **给定** 用户正在大图模式查看截图，**当** 放大并平移截图，**则** 用户可以查看截图细节，关闭后回到原功能和原截图状态。

---

### 用户故事 2 - 在手机上更快扫读功能与路线结果（优先级：P1）

手机用户浏览首页时，希望功能介绍不要像长说明书一样一项项纵向占满页面，也希望在线试查的路线结果卡片更紧凑，能一眼看到路线号、候车状态、站点和车费、耗时、步行距离。

**优先级原因**：手机端是普通用户访问主页和试查路线的高频场景；过长的功能介绍和过高的路线卡片会阻碍用户继续了解、试用和下载。

**独立测试**：在 390px 手机视口打开首页，查看功能介绍区和在线试查结果列表；确认功能卡片为 2 列紧凑展示，路线卡片中标签和值清楚区分。

**验收场景**：

1. **给定** 用户在手机端查看功能介绍区，**当** 页面展示 6 个主要功能，**则** 功能以 2 列紧凑卡片展示，且桌面端功能介绍布局不受影响。
2. **给定** 后续增加到最多 10 个主要功能，**当** 用户在手机端浏览功能介绍区，**则** 页面仍按 2 列扩展，不退回单列长列表，也不依赖横向滑动才能发现主要功能。
3. **给定** 用户在手机端查看路线结果，**当** 某条路线缺少站点名称，**则** 卡片以简洁提示表达站点资料暂未返回，不继续用“上车站 / 下车站”占据大量空白。
4. **给定** 用户在手机端查看路线结果，**当** 查看车费、耗时和步行信息，**则** 三个项目在同一信息行内展示，项目名称和值有明显视觉层级差异。

---

### 用户故事 3 - 通过更自然的功能文案理解车费比较（优先级：P2）

用户浏览功能介绍或轮播文案时，希望看到面向产品用户的自然说明，而不是开发修改要求原文。关于路线比较中的费用信息，用户应理解这是帮助选择路线的关键因素之一，而不是看到内部需求描述。

**优先级原因**：文案直接影响用户对功能价值的理解；内部需求原文出现在页面上会显得粗糙、不可信。

**独立测试**：切换 `zh-Hant`、`zh-Hans` 和 `en`，查看功能介绍和轮播中与费用比较相关的标题、说明、图片 alt 和无障碍文案。

**验收场景**：

1. **给定** 用户查看费用相关功能介绍，**当** 页面显示功能标题，**则** 标题使用“車費一眼看清 / 车费一眼看清 / Fare at a glance”语义。
2. **给定** 用户查看费用相关说明，**当** 页面显示配套描述，**则** 文案说明同页比较车费、行程时间和步行距离的用户价值，不出现内部修改要求原句。
3. **给定** 用户切换三种语言，**当** 查看同一功能说明，**则** 三语表达同一事实，`zh-Hant` 和 `en` 均为自然文案，不是机械直译。

### 边界情况

- 当某个功能只有一张截图时，页面内截图区和大图模式都不得显示“还有更多截图”的暗示或不可用切换控件。
- 当用户在截图区拖动时，系统不得误切换整个功能；当用户在文字介绍区拖动时，系统不得误切换同功能截图。
- 当用户在大图模式中缩放到大于初始尺寸后，左右平移必须优先查看当前图片细节，不能因平移被误判为切换功能。
- 当用户使用键盘或读屏访问截图和大图模式时，系统必须提供可理解的名称、关闭方式和同功能截图切换方式。
- 当用户开启减少动态效果时，页面应减少非必要动画，但仍能手动打开大图、切换功能和切换同功能截图。
- 当手机端功能标题在英文或繁体中文下较长时，卡片文字不得溢出或互相重叠；必要时允许单卡内部合理换行。
- 当手机路线结果中的英文标签或较长站点名过长时，卡片不得横向溢出；站点信息可以截断或换行，但车费、耗时和步行标签值必须可辨识。
- 当用户期待完整出行路线规划、港铁、铁路、渡轮或其他交通工具查询时，本次优化不得扩大网站承诺，页面仍必须保持香港巴士和 Citybus / 城巴试查边界。
- 当后续阶段涉及服务端 HTTP API 变更时，必须另行维护 OpenAPI 3.1 YAML；本功能本身不触发 API 变更。
- Figma 节点已于 `2026-06-30` 按用户提供的节点报告回填；后续如果 Figma 节点被复制、删除或重命名，必须同步更新本规格、`figma.md` 和相关契约，不能继续引用失效节点。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须在桌面端将首页 hero 右侧截图组调整为更醒目的中等放大比例，主截图和后排堆叠图必须同等比例放大。
- **FR-002**：系统必须在桌面端保留 hero 功能说明文字的可读宽度，不得因为放大截图导致标题、说明或圆点导航拥挤、重叠或异常换行。
- **FR-003**：系统不得在生产页面显示独立的放大提示器、悬浮放大图标或“点击放大”教学文案。
- **FR-004**：用户必须能够点击主截图打开大图查看模式。
- **FR-005**：大图查看模式必须支持放大、缩小、放大后平移、关闭，以及同一功能截图组内的上一张/下一张查看。
- **FR-006**：大图查看模式必须只切换同一功能内的截图，不得通过左右滑动跨到其他功能场景。
- **FR-007**：系统必须把页面内截图展示区和文字介绍区作为不同手势目标：截图区滑动或拖动只切换同功能截图，文字介绍区滑动或拖动切换整个功能。
- **FR-008**：场景圆点必须继续作为功能导航控件，点击后切换到对应功能场景。
- **FR-009**：后排堆叠截图必须保持可点击，用于切换当前功能内的主截图。
- **FR-010**：手机端功能介绍必须采用 2 列紧凑卡片展示，不使用分组胶囊，不退回单列长列表。
- **FR-011**：手机端功能卡必须尽量压低单卡高度、内边距和行间距，同时保留图标、标题和一句短说明。
- **FR-012**：桌面端功能介绍布局必须保持现状不变，不因手机端压缩方案而重排。
- **FR-013**：手机端路线结果卡片必须降低路线号的视觉重量，不再使用过粗过大的第一行路线号。
- **FR-014**：手机端路线结果卡片必须在第一行同时展示路线号和候车状态，候车状态应易于扫读但不抢占主要信息。
- **FR-015**：手机端路线结果卡片必须在站点资料缺失时展示简洁缺失提示，不得继续用“上车站 / 下车站”空占位拉高卡片。
- **FR-016**：手机端路线结果卡片必须把车费、耗时和步行合并到同一信息行或同一紧凑指标带中。
- **FR-017**：车费、耗时和步行必须同时展示项目名称和值，项目名称和值必须有明显样式层级差异。
- **FR-018**：费用相关功能标题必须改为 `車費一眼看清`、`车费一眼看清` 和 `Fare at a glance`。
- **FR-019**：费用相关说明必须表达“同页比较车费、行程时间和步行距离以辅助选择路线”的用户价值，不得使用内部修改要求原句。
- **FR-020**：所有新增或修改的用户可见文字、图片 alt、aria 文案和状态提示必须覆盖 `zh-Hant`、`zh-Hans` 和 `en`。
- **FR-021**：系统必须保持当前网站范围边界，不得因本次文案或 UI 优化暗示支持完整出行路线规划、港铁、铁路、渡轮或其他非香港巴士查询。
- **FR-022**：本功能必须在后续阶段产出或更新 Figma 设计，并记录文件链接、关键节点、桌面/手机 viewport、交互状态和版本说明。
- **FR-023**：系统不得因本功能新增、修改或移除服务端 HTTP API。

### 关键实体（涉及数据时填写）

- **功能展示截图组**：首页 hero 中归属于某个功能场景的一组 App 截图。关键属性包括当前功能、默认主截图、同功能其他截图、堆叠展示状态、大图查看状态和同功能切换范围。
- **手势区域**：用户在 hero 中操作的交互目标。关键属性包括截图区、文字介绍区、触发动作、切换目标和误触防护。
- **大图查看会话**：用户点击截图后进入的图片查看状态。关键属性包括当前截图、缩放比例、平移位置、同功能截图索引、关闭方式和无障碍焦点范围。
- **手机功能卡**：移动端功能介绍的紧凑卡片。关键属性包括图标、三语标题、三语短说明、卡片顺序和扩展到 10 个功能时的布局位置。
- **路线结果卡片**：在线试查返回的候选路线展示单元。关键属性包括路线号、候车状态、上车站、下车站、站点缺失提示、车费、耗时和步行距离。
- **费用功能文案项**：解释费用比较能力的三语文案。关键属性包括标题、说明、适用区域、香港繁体审校状态和英文自然表达状态。
- **Figma 设计引用**：后续阶段引用的设计源。关键属性包括目标文件链接、计划页面、关键节点、节点 ID、覆盖 viewport 和交互状态。

## 成功标准（必填）

### 可衡量结果

- **SC-001**：在 1440px 桌面视口中，hero 右侧主截图视觉尺寸比当前生产版本至少提升约 20%，后排堆叠图同步放大，且右侧文字不重叠、不被挤出可读范围。
- **SC-002**：桌面 hero 生产页面中 100% 不出现独立放大提示器、悬浮放大图标或“点击放大”教学文案。
- **SC-003**：用户点击主截图后，能在不离开当前页面的情况下完成打开大图、放大、平移、切换同功能截图和关闭。
- **SC-004**：在页面内截图区执行一次有效左右拖动时，只切换同功能截图；在文字区执行一次有效左右拖动时，只切换整个功能。
- **SC-005**：在 390px 手机视口中，6 个功能介绍以 2 列、3 行形式呈现；扩展到 10 个功能时仍以 2 列扩展，不需要横向滑动才能发现主要功能。
- **SC-006**：在 390px 手机视口中，路线结果列表首屏能看到首个完整候选路线卡片和第二个候选路线的主要入口信息。
- **SC-007**：手机路线结果卡片中，车费、耗时和步行三项 100% 同时显示项目名称和值，且项目名称和值样式可区分。
- **SC-008**：费用相关标题和说明在三种语言中 100% 覆盖，不出现“多程总车费一眼看清”或用户内部修改要求原句。
- **SC-009**：本功能的规格、设计记录和后续计划能定位到稳定 HTML 原型、目标 Figma 文件和 6 个真实 Figma 节点 ID。
- **SC-010**：本功能不产生任何服务端 HTTP API 新增、修改或删除；既有 OpenAPI 契约保持不变。
