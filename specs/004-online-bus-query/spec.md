# 功能规格：在线巴士路线查询

**功能标识**：`004-online-bus-query`

**创建日期**：2026-06-16

**状态**：Draft

**输入**：用户描述："实现在线查询功能：用户从网站主页点击「在线查询」或滚动到查询区后，可以输入起点和终点，从地点检索候选中选择地点，点击查询后查看真实香港巴士路线结果。前端地点查询和路线查询均通过服务端接口实现。结果按参考截图展示路线号、起始站点、下车站点、价格、耗时、步行距离和等候时间；不需要多班 ETA、监控功能、排序 UI 或卡片详情展开。页面布局必须先讨论确认后再实现。"

## Clarifications

### Session 2026-06-16

- Q: Web 在线查询是否接入真实 Citybus 数据源？ → A: 是，做真实 Citybus P2P 查询 MVP，但不保存路线、不做监控、不做多班 ETA、不做排序 UI。
- Q: Citybus `l` 参数如何映射语言？ → A: `zh-Hant` 使用 `l=0`，`en` 使用 `l=1`，`zh-Hans` 使用 `l=2`；已用真实接口验证地点和路线摘要。
- Q: `showstops2.php` 的简体站名如何处理？ → A: `showstops2.php` 用于定位 stop id；简体站名不足时用 DATA.GOV.HK stop API 的 `name_sc` 补齐。
- Q: 页面布局采用哪种方案？ → A: 采用 A 方案：桌面保留左侧说明、右侧真实查询工具，移动端堆叠。
- Q: 查询结果是否包含 Citybus P2P 返回的巴士换乘路线？ → A: 包含纯巴士换乘路线，不扩展到地铁、铁路、渡轮或完整跨交通工具规划。
- Q: 路线结果默认返回多少条？ → A: 后端最多返回 20 条，默认按耗时排序，前端全部展示并允许滚动查看。
- Q: 地点候选展示多少条？ → A: 后端最多返回 100 条；前端下拉接收全部候选，面板可滚动，可视高度约 8-10 条。
- Q: 默认是否预选地点或展示旧静态结果？ → A: 不预选，旧静态示例删除，初始空态提示用户选择起点和终点。
- Q: 路线查询和 ETA 查询是否拆分？ → A: 拆分为路线摘要接口和批量 ETA 接口；ETA 一次调用查询全部返回路线。
- Q: 结果卡是否显示铃铛、支持详情展开、排序或刷新？ → A: 不显示铃铛，不展开详情，不做排序 UI，不做独立刷新按钮。
- Q: 用户切换语言时如何处理当前结果？ → A: 同一组已选起终点自动重查路线和 ETA；失败保留旧结果并轻量提示。若起终点已变更，新查询失败不展示旧路线。
- Q: API 路径如何命名？ → A: 统一使用 `/api/routes/query_places`、`/api/routes/query_routes`、`/api/routes/query_etas`。
- Q: 业务参数和 requestId 放在哪里？ → A: 在线查询相关 API 请求和响应业务字段均使用 JSON body；响应统一 envelope：`{ requestId, data, error }`。二进制下载接口可例外。
- Q: 前端是否提交裸经纬度？ → A: 不提交。地点检索返回 `placeToken`，路线查询只接收起终点 token。
- Q: token 有效期？ → A: `placeToken` 15 分钟，`etaToken` 5 分钟，均需要签名防篡改。
- Q: 日志记录哪些内容？ → A: 记录起终点名称、经纬度、语言、关键节点、关键结果、耗时、结果数量、缓存命中、限流和错误类型；不记录 Cookie、token、完整外部 URL、第三方原始响应或 HTML。日志输出 stdout，部署层固定保留 7 天。

## 用户场景与测试（必填）

### 用户故事 1 - 选择起终点并查看路线结果（优先级：P1）

香港巴士用户访问网站后，希望直接试用基础巴士路线查询：输入起点和终点关键词，从候选地点中选择，点击查询后看到可比较的路线结果。

**优先级原因**：在线查询是网站核心范围之一，也是用户在下载 App 前判断查询能力是否可信的关键路径。

**独立测试**：打开首页在线查询区，输入并选择起点和终点，点击查询后看到最多 20 条真实香港巴士路线结果，结果卡展示路线号、上车站、下车站、价格、耗时、步行距离和候车状态。

**验收场景**：

1. **给定** 用户进入在线查询区且未选择地点，**当** 查看查询工具，**则** 起点和终点为空，结果区提示选择起终点后查询。
2. **给定** 用户在起点输入框输入关键词，**当** 地点检索成功，**则** 下拉列表展示当前语言地点名称，用户可以从候选中选择地点。
3. **给定** 用户已从候选中选择起点和终点，**当** 点击查询，**则** 结果区显示 loading，并在成功后展示按耗时排序的路线卡片列表。
4. **给定** 路线结果包含换乘巴士路线，**当** 用户查看结果卡，**则** 路线号以链式形式展示，并以第一段上车站和最后一段下车站作为站点摘要。

---

### 用户故事 2 - 查看首程候车状态（优先级：P2）

用户查看路线结果时，希望看到每条路线的首程候车状态，帮助判断是否适合当前出发。

**优先级原因**：首程 ETA 是 Android App 的核心结果信息之一，但它不应阻塞路线摘要展示。

**独立测试**：路线摘要返回后，页面立即展示结果卡并显示候车查询中；批量 ETA 返回后，各卡片更新为等候分钟、即将到站或候车暂不可用。

**验收场景**：

1. **给定** 路线摘要已返回但 ETA 尚未返回，**当** 用户查看结果卡，**则** 候车区域显示查询中状态，路线摘要仍可读。
2. **给定** 批量 ETA 查询成功，**当** 页面合并响应，**则** 对应路线卡显示当前语言的首程候车状态。
3. **给定** 批量 ETA 查询失败、token 过期或单条 ETA 不可用，**当** 页面处理响应，**则** 路线结果保留，未成功卡片显示候车暂不可用。

---

### 用户故事 3 - 语言切换后保持查询一致（优先级：P2）

繁体、简体和英文用户在查询过程中切换网站语言时，需要看到与当前语言一致的地点、站点、状态和错误文案，同时不丢失已选择的查询上下文。

**优先级原因**：三语切换是项目核心能力；动态外部查询结果也必须遵守三语体验。

**独立测试**：用户选择同一组起终点并完成查询后，切换 `zh-Hant`、`zh-Hans` 或 `en`，页面自动重新查询当前起终点并展示当前语言结果；如果重查失败，旧结果仍保留。

**验收场景**：

1. **给定** 用户已选择起点和终点，**当** 切换语言，**则** 已选地点的输入框显示名尽量切换到当前语言；无法回填时保留旧显示名，不清空选择。
2. **给定** 用户已看到路线结果，**当** 切换语言且重新查询成功，**则** 地点、站名、价格/耗时/步行标签和候车状态按当前语言展示。
3. **给定** 用户切换语言后重新查询失败，**当** 页面处理失败，**则** 保留旧结果，并以轻量状态提示仍显示上次结果。

---

### 用户故事 4 - 理解范围和降级状态（优先级：P3）

用户需要明白网页查询只是基础香港巴士试用，保存路线、监控、多班 ETA 和更多详情仍需下载 App；当外部服务失败或没有路线时，页面需要给出可信状态而不是误导。

**优先级原因**：清楚的范围边界可以避免把网站误解为完整出行规划工具，也能在外部服务波动时保持信任。

**独立测试**：查看在线查询区说明、0 结果空态、外部服务失败态、限流态和 token 过期态；确认文案三语完整且不暗示非巴士交通或完整路线规划。

**验收场景**：

1. **给定** 用户查看在线查询区说明，**当** 阅读左侧文案，**则** 明白网页提供基础香港巴士路线试用，完整保存、监控和更多详情请下载 App。
2. **给定** 查询成功但没有路线，**当** 页面展示结果区，**则** 用户看到正常空态，并被引导尝试附近地点或调整起终点。
3. **给定** 外部服务失败、超时或解析失败，**当** 页面展示错误态，**则** 用户看到可重试提示；如果起终点已变更，不展示旧路线结果。

### 边界情况

- 当用户只输入文字但未从候选中选择地点时，系统不得允许路线查询，必须提示从列表中选择地点。
- 当起点和终点为同一个候选地点时，系统不得允许查询，必须展示字段级错误。
- 当地点检索返回超过可视区域时，下拉面板必须保持固定最大高度并允许滚动，不得撑开手机屏幕。
- 当地点检索失败时，当前已选地点不得被意外清空；用户可以继续编辑或重试。
- 当 `placeToken` 过期、篡改或不属于候选地点时，路线查询必须失败并提示重新选择地点。
- 当路线查询返回 0 条路线时，系统必须显示正常空态，不得把它当成外部服务失败。
- 当路线查询外部服务失败、超时、被限流或解析失败时，系统必须显示可重试错误态，并记录可观测日志。
- 当批量 ETA 查询失败时，系统必须保留路线摘要结果，不得清空列表。
- 当用户快速连续查询或切换语言时，旧响应不得覆盖新查询结果。
- 当用户切换 `zh-Hant`、`zh-Hans`、`en` 时，当前页面状态、已选地点和结果展示必须尽量保持一致；失败时保留旧结果并轻量提示。
- 当用户期待完整出行路线规划、地铁、铁路、渡轮或其他非香港巴士查询时，页面必须表达本功能仅限香港巴士路线试用。
- 当用户在手机和电脑之间切换访问时，查询输入、下拉列表、交换按钮、查询按钮、loading、结果卡和错误状态都必须完整可见、可读、可操作。
- 当前端 UI 进入计划阶段前，必须能定位对应 Figma 文件、桌面/移动节点和交互状态；若 Figma 节点缺失，计划阶段不得视为视觉门禁通过。
- 当服务端内部错误、panic 或外部依赖异常发生时，系统必须恢复为受控响应，记录 requestId 和脱敏上下文，并避免服务整体崩溃。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须把现有在线查询区从静态演示升级为真实香港巴士路线试用查询。
- **FR-002**：系统必须保留首屏“在线查询”行动，点击后跳转到 `#online-query` 查询区域。
- **FR-003**：在线查询区桌面布局必须采用左侧说明、右侧查询工具；移动端必须改为上下堆叠。
- **FR-004**：在线查询说明必须表达网页提供基础香港巴士路线试用，保存路线、监控和更多详情需要下载 App。
- **FR-005**：起点和终点默认必须为空；系统不得展示旧静态示例路线作为默认结果。
- **FR-006**：用户输入至少 1 个字符后，系统必须延迟触发地点检索，并只保留当前输入的最新候选响应。
- **FR-007**：地点候选必须来自服务端地点检索接口，最多 100 条；前端下拉必须可滚动，可视高度约 8-10 条。
- **FR-008**：地点候选只展示当前语言的一种名称，不展示双语副名。
- **FR-009**：用户必须从地点候选中选择起点和终点后才能查询路线；系统不得允许仅凭自由文本查询路线。
- **FR-010**：系统必须阻止起点和终点为同一候选地点的查询。
- **FR-011**：系统必须支持交换起点和终点；交换后清空旧路线结果，用户需重新查询。
- **FR-012**：地点检索接口必须返回可用于后续查询的受保护地点 token；路线查询接口不得要求前端提交裸经纬度。
- **FR-013**：地点 token 必须有 15 分钟有效期，并能防止篡改或伪造。
- **FR-014**：路线查询接口必须根据已选起点和终点返回最多 20 条香港巴士路线结果，默认按耗时排序。
- **FR-015**：路线结果必须包含 Citybus P2P 返回的纯巴士换乘路线，但不得扩展到非巴士交通或完整出行规划。
- **FR-016**：每条路线结果必须包含路线标识、运营商、路线号链、上车站名、下车站名、港币价格数值、耗时分钟数、步行距离米数和可选 ETA token。
- **FR-017**：多段巴士路线必须用结构化路线号链展示；若结构化路线号不可用，系统可以使用后端返回的路线名称作为后备。
- **FR-018**：前端必须根据当前语言格式化价格、耗时和步行距离；后端不得把这些 UI 文案作为唯一主契约。
- **FR-019**：结果卡必须按用户截图的信息层级展示路线号、站点摘要、候车状态、价格、耗时和步行距离。
- **FR-020**：结果卡不得显示铃铛，不得提供监控入口。
- **FR-021**：结果卡不得在本功能中提供详情展开、多班 ETA、排序控件或独立刷新按钮。
- **FR-022**：路线查询成功后，前端必须一次性调用批量 ETA 接口查询全部返回路线的首程候车状态。
- **FR-023**：ETA token 必须有 5 分钟有效期，并能防止篡改或伪造。
- **FR-024**：批量 ETA 接口必须返回与路线结果可合并的首程候车状态；单条失败不得导致路线结果被清空。
- **FR-025**：ETA 未返回前，结果卡必须显示候车查询中；ETA 失败、过期或不可用时，结果卡必须显示候车暂不可用。
- **FR-026**：系统必须为每次查询使用 requestId；旧路线或 ETA 响应不得覆盖新的查询结果。
- **FR-027**：语言切换时，系统必须对同一组已选起终点自动重新查询路线和 ETA；重查失败时必须保留旧结果并轻量提示。
- **FR-028**：用户已改变起点或终点后，新查询失败不得展示旧路线结果。
- **FR-029**：路线结果为 0 条时，系统必须展示正常空态，引导用户尝试附近地点或调整起终点。
- **FR-030**：外部服务失败、超时、解析失败、限流或 token 错误时，系统必须展示可恢复错误状态，不得编造实时数据。
- **FR-031**：在线查询 API 必须使用 `POST` 和 JSON body 承载业务参数；业务响应必须使用 `{ requestId, data, error }` JSON envelope。二进制下载接口不受此规则影响。
- **FR-032**：在线查询 API 路径必须为 `/api/routes/query_places`、`/api/routes/query_routes`、`/api/routes/query_etas`。
- **FR-033**：HTTP 状态码必须按标准语义使用；业务错误细节必须放入 JSON body 的 `error` 字段。
- **FR-034**：前端用户可见错误文案必须按 `error.code` 做三语翻译，后端 message 不得直接作为 UI 文案展示。
- **FR-035**：系统必须为地点检索、路线查询和 ETA 查询提供轻量限流；超过限制时返回可恢复错误。
- **FR-036**：后端必须输出结构化日志，覆盖请求入口、地点检索、路线查询、站点预览、ETA token、批量 ETA、缓存、限流、错误映射、关键结果和总耗时。
- **FR-037**：后端日志可以记录起终点名称和经纬度，但不得记录 Cookie、token、完整外部 URL、第三方原始响应、HTML 或不可控大段内容。
- **FR-038**：部署层必须固定保留查询服务 stdout 日志 7 天；应用内不实现文件轮转或保留期配置。
- **FR-039**：服务端不得以 panic 作为业务控制流，HTTP 入口必须启用 recovery 和请求日志。
- **FR-040**：系统必须在计划阶段前提供 Figma 在线查询 v2 桌面与移动设计节点，覆盖主要交互状态。

### 关键实体（涉及数据时填写）

- **地点候选**：用户输入关键词后由 Citybus 返回并由后端归一化的可选择地点。关键属性包括当前语言名称、坐标、受保护地点 token 和过期时间。
- **地点 token**：后端签发的短期查询凭证，用于证明前端路线查询来自已选候选地点，内部关联地点名称、坐标、语言和过期时间。
- **路线查询请求**：一次由起点 token、终点 token、语言和 requestId 组成的路线试用查询。
- **路线选项**：可展示的一条香港巴士路线结果。关键属性包括路线号链、运营商、上车站、下车站、价格、耗时、步行距离、排序位置和 ETA token。
- **ETA token**：后端签发的短期候车查询凭证，用于批量查询首程 ETA，内部关联路线结果和首程 ETA 查询所需信息。
- **候车状态**：单条路线的首程候车展示状态，包括查询中、等候分钟、即将到站和候车暂不可用。
- **查询错误**：由参数无效、token 过期、限流、外部服务失败、解析失败或系统异常产生的可映射错误。
- **查询日志事件**：用于服务观测的结构化日志记录，包含 requestId、语言、阶段、耗时、结果数量、缓存命中、起终点信息和错误类型。

## 成功标准（必填）

### 可衡量结果

- **SC-001**：用户在桌面或手机在线查询区中，能在选择两个地点并点击一次查询后看到真实香港巴士路线结果或明确空态。
- **SC-002**：地点候选必须在三种语言下展示当前语言名称；`zh-Hant`、`zh-Hans` 和 `en` 均能完成地点选择和路线查询。
- **SC-003**：查询成功时，95% 的路线摘要请求在正常外部服务可用条件下 8 秒内完成并显示结果列表。
- **SC-004**：路线结果最多展示 20 条，并默认按耗时升序排列。
- **SC-005**：批量 ETA 失败时，100% 情况下路线摘要结果仍保留，未成功路线显示候车暂不可用。
- **SC-006**：语言切换重查失败时，100% 情况下旧结果仍保留；起终点变更后的新查询失败时，100% 情况下不展示旧路线。
- **SC-007**：桌面 1440px 和手机 390px 验证中，地点下拉、交换按钮、查询按钮、loading、结果卡和错误/空态均可见、可读、可操作。
- **SC-008**：用户不能通过自由文本、过期 token、篡改 token 或相同起终点发起成功路线查询。
- **SC-009**：后续实现者能在 2 分钟内从 OpenAPI 文档定位 3 个在线查询接口、request/response envelope、错误格式、示例和中文 API UI。
- **SC-010**：开发者能通过同一个 requestId 在结构化日志中串联请求入口、外部调用、缓存/限流、路线结果、ETA 结果和错误映射。
- **SC-011**：页面和 FAQ 不出现完整出行路线规划、非香港巴士交通查询、监控、多班 ETA、排序 UI 或路线详情展开的当前可用暗示。

## 假设

- 网站在线查询只作为基础试用功能，不替代 Android App 的完整功能。
- Citybus mobile 与 DATA.GOV.HK 在大多数时间可访问，但其格式和可用性可能变化，因此解析和降级必须可测试。
- 记录起终点名称和经纬度按本产品定义不视为个人资料；但日志仍需避免记录 Cookie、token、第三方原始响应和不可控大段内容。
- 地点和站点名称以外部服务当前语言返回为准；项目自写 UI 文案进入前端三语资源。
- 简体站名优先从 DATA.GOV.HK stop API 的 `name_sc` 补齐；补齐失败时可使用可用的站点显示名并记录日志。
- Figma MCP 暂不可写入时，已通过本地 Figma 开发插件在现有首页 Figma 文件中创建 Online Query v2 节点。
- 不引入数据库；短期缓存、限流和 token 校验不需要持久化用户查询。
- 后端输出结构化 stdout 日志，7 天保留由部署层负责。
