# 第 0 阶段研究：首页 UI 优化

## 决策 1：沿用 React + Vite 静态内容模型，扩展为首页 v2 内容契约

**Decision**：继续以 `frontend/src/content/*` 作为首页内容来源，扩展 `CarouselSlide` 为更明确的“核心功能展示项 + 截图图集”模型，并用 feature contract 约束 4 个核心功能点、三语文案和截图资产。

**Rationale**：现有项目已经有 `homepageContent.ts`、`carouselSlides.ts`、`sectionsContent.ts`、`types.ts` 和 Ajv 内容契约测试。沿用该模式可以保持前端内容可测试、可审查，也能让 `zh-Hant`、`zh-Hans`、`en` 的完整性继续被单元测试覆盖。

**Alternatives considered**：
- 在组件内直接硬编码 4 个功能点：实现快，但违反 i18n 和内容契约约束。
- 引入 CMS 或远程配置：当前只是首页静态优化，复杂度过高，也不利于截图资产审查。

## 决策 2：原始截图不进入前端，使用脱敏副本和 manifest

**Decision**：将 `app真实截图/` 仅视为原始输入；实施阶段生成脱敏副本到 `frontend/src/assets/app-screenshots/real/`，并维护截图资产 manifest。manifest 记录原始分组、主图、输出路径、脱敏方式、可展示状态和保留数值说明。

**Rationale**：用户要求使用真实截图，同时要求替换真实地点、巴士路线等真实信息，并模糊手机无关内容；又明确价格、时间和 ETA 不需要替换。manifest 能把这类例外沉淀为可审查资产契约，避免实现阶段靠口头记忆。

**Alternatives considered**：
- 直接复制原图到前端：会泄露真实地点、路线号、搜索记录和系统无关内容。
- 用现有 SVG mock 继续占位：不能满足“真实截图”和“高级可信展示”的目标。
- 截图全量高斯模糊：安全但损害用户理解真实数据形态，且会误伤价格、时间和 ETA。

## 决策 3：外层 4 个核心功能点自动轮播，内层图集只手动切换

**Decision**：首页展示模块分两层状态：外层 `activeFeatureId` 每 4.5 秒自动切换一次，动画 600-700ms，使用深度换层/堆叠层级变化；内层 `activeScreenshotId` 只响应用户手动选择，不自动轮播、不提供左右箭头。

**Rationale**：用户已明确“不是同一功能点下的多张堆叠图自动轮播”，而是“4 个核心功能点自动轮播”。分层状态能保持高级感和信息节奏，同时避免用户查看同组截图时被打断。

**Alternatives considered**：
- 所有截图统一一个 carousel：会违反用户确认的交互模型。
- 只提供静态展示：不能满足首屏右侧自动轮播和动画要求。
- 左右箭头轮播：用户明确不要左右箭头；可保留非箭头式分页点或当前项指示。

## 决策 4：自动轮播必须可暂停、可降级，并保持状态

**Decision**：外层轮播在 hover、focus、pointer/touch 交互期间暂停；用户离开或交互结束后恢复。`prefers-reduced-motion: reduce` 下停止自动动画或将动画降级为无位移/淡入，仍允许用户查看 4 个核心功能点。语言切换不得重置 `activeFeatureId` 或 `activeScreenshotId`。

**Rationale**：规格要求用户查看截图或文字时不被打断，也要求语言切换保持稳定。减少动态效果是可访问性底线，避免给用户带来眩晕或阅读干扰。

**Alternatives considered**：
- 轮播一直自动播放：会打断用户查看堆叠截图。
- 语言切换后回到第一项：实现简单，但违反边界情况。
- 减少动态效果下隐藏整个展示模块：会降低内容覆盖，不符合可用性要求。

## 决策 5：`zh-Hant` 独立按香港书面语重写

**Decision**：`zh-Hant` 文案单独编写，语气保持自然、清楚、可信，不过度口语化；不得从 `zh-Hans` 做机械字形转换。范围类文案必须明确“城巴 / Citybus”，并清楚说明不支持九巴、港铁、铁路、渡轮、其他交通工具或完整出行规划。

**Rationale**：目标用户是香港本地及可能用繁体中文浏览的用户。用户明确指出不能只做单词替换，需要考虑整句语气和说话习惯。独立文案也能避免把“香港巴士”误写为覆盖所有运营商。

**Alternatives considered**：
- 使用简中转换繁中：速度快，但用户已经否定，且不符合香港语境。
- 过度口语化港式表达：可能影响可信度和正式感。
- 只在 FAQ 说明城巴范围：首屏和功能区仍可能误导用户。

## 决策 6：前后端监听所有本机网络接口，正式前端支持标准 HTTP 入口

**Decision**：前端开发和预览命令应支持 `0.0.0.0` 监听；后端 Go 服务应支持通过配置监听所有本机接口，默认满足同局域网访问。本轮不改变 API 路径。正式部署时前端可通过服务自身、反向代理或端口映射提供标准 HTTP 入口，计划和 quickstart 只约束访问结果。

**Rationale**：用户明确要求前后端不要只监听本地回环地址，并补充正式部署时前端支持正式 HTTP 端口。现有 `frontend/package.json`、`frontend/vite.config.ts`、`backend/cmd/server/main.go` 都显式使用 `127.0.0.1`，需要纳入实现范围。

**Alternatives considered**：
- 只在命令行临时传 `--host 0.0.0.0`：容易被脚本默认值覆盖，验收不稳定。
- 只改前端监听：手机访问首页后下载 API 仍可能访问失败。
- 强制前端进程直接监听 80 端口：在非 root 环境不稳定；标准 HTTP 入口可以由反向代理或端口映射提供。

## 决策 7：不新增或修改服务端 HTTP API，OpenAPI 只做回归保护

**Decision**：本轮不创建新的 OpenAPI 文件，不修改下载 endpoint、响应头、错误格式或下载业务语义。实现阶段只运行既有 OpenAPI lint/bundle 或等价检查，确认下载契约没有被监听地址配置误伤。

**Rationale**：规格明确不得修改既有 Android APK 下载 API 路径或下载后端业务语义。监听地址属于部署/入口配置，不是 API 合同变化。

**Alternatives considered**：
- 为首页内容新增 API：当前内容是静态首页数据，不需要后端服务。
- 修改下载 API 增加 UI 所需字段：APK 版本和大小已有 manifest/既有下载事实来源，本轮不应扩大后端语义。

## 决策 8：验证以“内容契约 + 浏览器截图 + 局域网访问”为主

**Decision**：单元测试覆盖内容和状态契约；Playwright 覆盖桌面、手机、下载直触发、自动轮播、手动图集、语言切换和截图证据；局域网访问用开发机 IP 进行手动或脚本验证；Figma 设计节点作为视觉源，浏览器截图作为实现证据。

**Rationale**：本功能风险集中在前端交互、文案、截图资产和双端体验。只跑构建不足以证明交互和视觉稳定性；只看 Figma 也不能证明真实浏览器可用。

**Alternatives considered**：
- 只靠人工浏览：缺少回归保障。
- 只靠单元测试：无法证明截图、布局、动画和下载行为在真实浏览器中可用。
- 只做桌面验证：规格明确要求手机和同局域网设备访问。
