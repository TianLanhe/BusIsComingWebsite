# UI 与内容需求质量检查清单：首页 UI 优化

**目的**：评审首页 UI 优化规格在交互、文案、素材、响应式、访问入口和宪法门禁上的完整性、清晰度与可衡量性，确保进入 `/speckit-plan` 前需求本身已经足够明确。
**创建日期**：2026-06-16
**功能**：[spec.md](../spec.md)

**说明**：本清单检查的是需求写作质量，不检查实现结果。若发现问题，应先更新 `spec.md` 或后续计划中的对应章节，再进入实现。

## 需求完整性

- [ ] CHK001 5 个用户故事是否都映射到可追踪的功能需求与成功标准，且没有遗漏下载、展示、图集、文案和访问入口任一主线？[Completeness, Spec §用户场景, Spec §FR-001-FR-026, Spec §SC-001-SC-013]
- [ ] CHK002 首屏下载、下载区详情卡、下载失败状态和 iPhone 暂未支持状态是否分别定义了需求边界，而不是混在同一个模糊下载描述中？[Completeness, Spec §FR-001-FR-005]
- [ ] CHK003 4 个核心功能点的外层自动轮播与同一功能点内截图图集的手动选择是否被分开定义，避免把两类轮播规则混淆？[Completeness, Spec §FR-006-FR-013]
- [ ] CHK004 真实截图脱敏是否覆盖真实地点、站名、路线号、搜索记录和手机系统无关内容，并明确价格、时间、ETA 数值属于保留例外？[Completeness, Spec §FR-014-FR-016]
- [ ] CHK005 城巴 / Citybus 口径和范围外能力说明是否完整覆盖九巴、港铁、铁路、渡轮、其他交通工具和完整出行规划？[Completeness, Spec §FR-017-FR-018]
- [ ] CHK006 `zh-Hant`、`zh-Hans` 和 `en` 是否覆盖所有新增或修改的用户可见文字，并明确繁中不是简中文字形转换？[Completeness, Spec §宪法对齐, Spec §FR-020-FR-021]
- [ ] CHK007 本地同一局域网访问和正式标准 HTTP 入口是否分别定义了用户结果与边界，且未把两者合并为单一“能访问”表述？[Completeness, Spec §FR-024-FR-025]

## 需求清晰度

- [ ] CHK008 “低权重”的 iPhone 暂未支持状态是否需要补充视觉优先级、位置或尺寸边界，避免在设计阶段被做得过强或过弱？[Ambiguity, Spec §FR-002]
- [ ] CHK009 “平滑的深度换层动画”是否已说明与时间、触发、暂停、恢复和减少动态效果相关的最低要求？[Clarity, Spec §FR-007, Spec §边界情况]
- [ ] CHK010 多图堆叠、主图、底层露出图和单图状态是否有清楚定义，能让计划阶段拆出稳定交互状态？[Clarity, Spec §FR-010-FR-013]
- [ ] CHK011 “不再套手机边框，但保留视觉边界”是否需要补充可接受的边界表达，例如圆角、描边、阴影或背景分隔的原则？[Ambiguity, Spec §FR-023]
- [ ] CHK012 “香港书面语偏自然、正式清楚、不过度口语化”是否具备可评审标准，能避免繁中仅做字符转换或变成广告腔？[Ambiguity, Spec §FR-020, Spec §SC-009, Spec §假设]
- [ ] CHK013 “标准 HTTP 入口”是否已清楚表达为访问结果，而不是绑定到某一种服务监听、反向代理或端口映射实现？[Clarity, Spec §FR-025, Spec §假设]

## 需求一致性

- [ ] CHK014 首屏直接下载的用户故事、功能需求、失败边界和成功标准是否一致，都排除了锚点跳转和平台分段框？[Consistency, Spec §用户故事 1, Spec §FR-001-FR-005, Spec §SC-001-SC-003]
- [ ] CHK015 外层 4 个核心功能点自动切换和内层截图图集不自动切换是否在用户故事、验收场景、功能需求和边界情况中保持一致？[Consistency, Spec §用户故事 2-3, Spec §FR-006-FR-013]
- [ ] CHK016 城巴范围说明是否与宪法中“香港巴士查询”定位兼容，并且没有把现阶段能力扩写成所有香港巴士公司？[Consistency, Spec §宪法对齐, Constitution §I, Constitution §IV]
- [ ] CHK017 “不修改下载 API 路径或后端业务语义”是否与局域网访问、正式 HTTP 入口和下载失败提示要求保持一致，没有引入隐含 API 变更？[Consistency, Spec §FR-024-FR-026, Spec §API 文档]
- [ ] CHK018 4 个核心展示点和下面 6 个功能卡片的定位是否区分清楚，避免计划阶段误把 6 个卡片改成外层轮播项？[Consistency, Spec §用户故事 2, Spec §FR-006, Spec §FR-019]

## 验收标准质量

- [ ] CHK019 下载、自动轮播、手动图集、截图脱敏、三语文案、城巴范围和访问入口的成功标准是否都有可客观判断的结果？[Acceptance Criteria, Spec §SC-001-SC-013]
- [ ] CHK020 截图脱敏的 100% 要求是否定义了足够具体的敏感信息类别，使后续评审能够判断截图是否可作为前端素材？[Measurability, Spec §SC-007]
- [ ] CHK021 保留价格、时间和 ETA 数值的成功标准是否与脱敏规则配对，避免审查时把这些数值误判为需要替换的信息？[Measurability, Spec §SC-008, Spec §FR-016]
- [ ] CHK022 繁中不是简中文字形转换的成功标准是否能通过抽查范围、示例或审核规则被客观评审？[Measurability, Spec §SC-009]
- [ ] CHK023 双端布局成功标准是否覆盖下载按钮、在线查询按钮、APK 元信息、iPhone 状态、截图堆叠和功能文案这些高风险元素？[Acceptance Criteria, Spec §SC-003, Spec §边界情况]

## 场景与边界覆盖

- [ ] CHK024 减少动态效果偏好、悬停、聚焦和点按期间暂停自动轮播的场景是否都有需求描述，而不是只出现在验收口头约定中？[Coverage, Spec §用户故事 2, Spec §边界情况]
- [ ] CHK025 语言切换后保持当前核心功能点和主图选择的状态保持要求是否足够明确？[Coverage, Spec §边界情况]
- [ ] CHK026 截图缺失、截图数量不符合假设或某组只有一张图时的需求是否足够覆盖计划阶段的素材处理边界？[Coverage, Spec §FR-013, Spec §假设]
- [ ] CHK027 键盘可达性、屏幕阅读器语义和焦点状态是否已作为需求覆盖外层轮播与内层截图选择？[Gap, Constitution §VII]
- [ ] CHK028 手机端触控目标、堆叠露出区域和小屏文字可读性的需求是否足够具体，避免移动端只能视觉可见但难以操作？[Gap, Spec §FR-022, Spec §SC-003]
- [ ] CHK029 同一局域网访问的安全边界是否有需求说明，例如仅用于本地验收、不暗示公开暴露开发服务？[Gap, Spec §用户故事 5]

## 依赖、假设与设计追踪

- [ ] CHK030 Figma v2 设计产物是否被定义为进入计划和实现前的必备依赖，并包含桌面、手机、下载状态、轮播状态和图集状态节点？[Dependency, Spec §Figma 设计, Constitution §VIII]
- [ ] CHK031 真实截图来源、4 个目录分组、序号 1 为主图和路线比较单图假设是否足够清楚，能支撑后续资产清单与脱敏记录？[Assumption, Spec §产品来源, Spec §假设]
- [ ] CHK032 Android 主项目、现有下载规格、下载 manifest 和本地截图之间的事实来源优先级是否明确，避免后续文案或素材与 App 当前能力冲突？[Dependency, Spec §宪法对齐, Constitution §I]
- [ ] CHK033 OpenAPI 不变更的边界是否足够明确，并说明若后续服务端 HTTP API 改动必须另开契约更新？[Boundary, Spec §API 文档, Spec §FR-026, Constitution §X]

## 歧义与冲突清理

- [ ] CHK034 “高级感”“优雅”“简洁”“自然”这类主观词是否都已落到可设计、可评审或可由 Figma 承接的具体要求？[Ambiguity, Constitution §VI, Spec §Figma 设计]
- [ ] CHK035 页面从“香港巴士”收窄到“城巴”后，在线查询、功能说明、FAQ 和下载介绍之间是否不会出现口径冲突？[Conflict, Spec §FR-017-FR-018]
- [ ] CHK036 开发监听所有本机网络接口与正式 HTTP 入口的需求是否区分了开发、验收和生产三个语境，避免计划阶段过度扩大网络暴露范围？[Ambiguity, Spec §FR-024-FR-025]
