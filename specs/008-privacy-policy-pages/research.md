# 第 0 阶段研究：隐私政策页面

**功能**：隐私政策页面  
**日期**：2026-06-30  
**输入**：[spec.md](./spec.md)、前期讨论结论、现有首页 SEO 与三语入口实现

## 决策 1：本轮只做网站隐私页，不修改 Android App

**决策**：新增 `/zh-hant/privacy/`、`/zh-hans/privacy/`、`/en/privacy/` 三个网站页面；Android App 代码、应用内跳转、Google Play Console 配置不在本轮实现范围。

**理由**：用户已确认“Android app 暂时先不改动，我们只做好网站的隐私页”。隐私页内容要覆盖 App 当前能力和官网试用查询事实，但交付边界属于网站。

**被拒绝方案**：同步修改 Android App 隐私入口。该方案会扩大测试面，也会把网站页面交付与 App 发布流程耦合。

## 决策 2：隐私页内容覆盖官网和 Android App

**决策**：页面标题与范围说明明确适用于 BusIsComing 网站和 Android App；默认 Google Play/App 隐私政策 URL 为 `https://www.busiscoming.com/zh-hant/privacy/`。

**理由**：未来 Android App 会跳转到网页隐私政策，且 Google Play Console 需要稳定公开 URL。用户确认主项目代码能力与官网下载 APK 能力保持一致，因此网站隐私页可以按当前主项目事实撰写。

**被拒绝方案**：只写网站隐私条款。该方案无法承担未来 Android App 隐私政策跳转和应用商店审核用途。

## 决策 3：内容保持简单，但澄清“不收集个人资料”的边界

**决策**：隐私页继续表达“不建立账户、不收集可识别身份资料、不做广告追踪、不出售资料”，同时说明为完成路线查询、定位换站、通知监测、语音提醒和网站试用查询，会处理路线、站点、设备权限状态、GPS 坐标或查询条件等功能必需的匿名信息。

**理由**：用户认为这些不是个人信息或个人资料，而是不包含用户信息的查询路线必需匿名信息。隐私页需要避免把首页“不收集个人资料”写成“不处理任何数据”，否则与定位、Google Geocoding、在线试查日志事实不一致。

**被拒绝方案**：逐项列出数据库字段或内部实现细节。用户确认不需要 DB 字段级枚举，隐私协议应尽量简单。

## 决策 4：明确外部服务，但不写不必要细节

**决策**：正文必须点名 Citybus、DATA.GOV.HK 和 Google Geocoding API；说明 GPS 坐标会在需要地点解析时发送给 Google Geocoding API。正文不写 Android 备份/设备迁移，不写 WeChat/Alipay/AlipayHK 包名检查，不写完整支付或钱包检测细节。

**理由**：前期讨论确认这些外部服务与坐标传输需要透明披露；同时用户要求内容简单，非必要内容不写入。

**被拒绝方案**：把所有第三方、系统权限和包检查都列入隐私政策。该方案会让页面过长，也会把非核心实现细节固化成对外承诺。

## 决策 5：信息架构采用“摘要卡 + 正文五章”

**决策**：页面首屏提供 4 个摘要卡：不建立账号身份、不做广告追踪或出售、收藏路线以设备本地为主、按功能需要连接外部服务。正文保持 5 个章节：适用范围、我们不收集什么、功能必需的信息如何处理、第三方服务、你的选择与联系我们。

**理由**：该结构兼顾官网访客快速浏览和 App 隐私政策页面的正式说明。摘要卡降低阅读成本，正文五章能覆盖合规事实但不啰嗦。

**被拒绝方案**：长篇法律条款式页面。用户明确要求简单，不要太啰嗦。

## 决策 6：隐私页不显示语言切换控件

**决策**：隐私页语言完全由 URL 决定，不显示当前首页的语言切换控件。未来 App 可按自身语言打开对应 URL，但网页隐私页内部不提供切换语言。

**理由**：用户确认“从手机进入到网页页面里不支持隐私政策页的语言切换”，并补充 App 未来可支持语言切换，但这不要求隐私网页内切换。

**被拒绝方案**：复用首页 Header 的语言切换器。该方案与用户确认的 App 跳转体验不一致。

## 决策 7：入口只放 footer，不进入主导航或 FAQ

**决策**：每个语言首页 footer 增加当前语言的隐私政策链接；不新增主导航入口，不新增 FAQ 入口。

**理由**：隐私政策属于信任与合规入口，不应与查询、下载、路线能力同级占用主导航。footer 是网站和 App 下载页常见位置。

**被拒绝方案**：加入顶部导航或 FAQ。该方案会放大隐私政策在主任务中的视觉权重。

## 决策 8：短期服务日志只做概括性披露

**决策**：网站在线试用查询可以说明会保留短期服务日志用于排障、稳定性和滥用防护，但不在用户可见文案里承诺具体天数。

**理由**：用户选择披露短期服务日志，但隐私协议内容要简单；具体保留期如果未来随运维策略调整，不应过早固定到用户页面。

**被拒绝方案**：公开写“7 天”。该方案会把内部可调整运维策略变成外部承诺。

## 决策 9：SEO 使用“页面类型 + 语言”的页面组模型

**决策**：SEO 配置从当前按语言首页扩展为 `home` 与 `privacy` 两个页面组。每个页面组拥有自己的 locale path、title、description、canonical、hreflang、Open Graph/Twitter 信息和 sitemap 条目。

**理由**：隐私页必须 canonical 到自身，且 hreflang 只能互指同一页面意图的三语版本。沿用首页 locale 模型会导致隐私页 canonical/hreflang 错误。

**被拒绝方案**：把隐私页 canonical 指向首页或只放进 sitemap 不配置页面级 SEO。该方案会削弱页面可索引性并制造重复内容信号。

## 决策 10：本功能不新增后端 API

**决策**：隐私政策页作为静态前端页面实现，不新增后端接口、不修改 OpenAPI、不新增 DDD bounded context。

**理由**：页面内容是静态政策信息，当前无需服务端动态配置。静态生成更利于 SEO 与部署稳定性。

**被拒绝方案**：通过后端 API 返回隐私政策内容。该方案增加不必要的运行时依赖和契约维护成本。

## 决策 11：Figma 先用本地插件 fallback 生成可追溯设计源

**决策**：计划产物包含 [figma.md](./figma.md) 和 `figma-plugin/`。如果当前会话无法直接写入目标 Figma 文件，后续可手工导入本地插件生成 `Privacy Policy Pages - 008` 页面，再回填真实节点信息。

**理由**：宪法要求涉及前端 UI 的 spec 有 Figma 文件或链接、关键节点、交互状态和版本说明。当前不能伪造真实 node ID，fallback 插件提供可复现的设计源。

**被拒绝方案**：只写文字计划或伪造 Figma 节点 ID。前者无法满足 UI 规格门禁，后者不可审计。

## 结论

后续实现应围绕一个静态、三语、可索引、footer 可达、无语言切换的隐私政策页面组展开。主要工程风险不在页面内容本身，而在现有 SEO 和静态生成模型需要从“语言首页”升级为“页面类型 + 语言”。
