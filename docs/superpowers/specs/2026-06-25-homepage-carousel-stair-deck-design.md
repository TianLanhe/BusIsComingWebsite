# 首页功能轮播牌堆交互设计

**日期**：2026-06-25  
**状态**：已获用户确认

## 原型是验收基准

本设计包含可打开的 HTML 原型：

- [2026-06-25-homepage-carousel-stair-deck.html](../prototypes/2026-06-25-homepage-carousel-stair-deck.html)

实现和验收必须以该原型为主要视觉基准，而不是只按文字解释。原型中的虚线底部基线是评审辅助层，用来证明后方图片没有低于主图底部；生产页面不得显示这条虚线或说明文字。

## 最终设计方向

同一功能场景内的多张截图采用“低旋转阶梯牌堆”展示。主图位于最上层，后方截图以轻微透明度、轻微缩放和小角度旋转从上方及左右露出，形成像一叠牌的层次，但不做大幅扇形展开。

桌面端后方卡片旋转角度为 `5deg`；手机端为避免横向占用过多，收敛到约 `4deg`。后方图片和主图以同一个底部视觉基线为准，后方图片必须整体上提，不能有角或边视觉上低于主图底部。

## 交互模型

功能场景切换和同场景图片切换是两个不同维度。

- 自动轮播只按功能场景切换，间隔仍为约 3 秒。
- 手动拖动或左右滑动只切换功能场景，不切换同场景图片。
- 场景点点是显式场景导航；点击第几个点就切换到对应功能场景。
- 同一功能场景内的多张图片只能通过点击牌堆中的图片切换主图。
- 点击后方露出的图片后，该图片升到最上层成为主图；原主图退到后方。
- 主图本身不做“点击循环下一张”，避免隐藏交互过多。

键盘和读屏用户仍需有可访问的场景切换方式。后方图片必须是真实可聚焦按钮或等价控件，具备清楚的 `aria-label`，例如“切换到常用路线截图 2”。

## 视觉约束

- 主图完整显示，保持当前手机截图比例和圆角质感。
- 后方截图最多显示两张；若当前场景只有一张图，只显示主图，不显示牌堆暗示。
- 后方截图透明度保持在能看清内容的范围，桌面建议 `0.70-0.78`，hover/focus 提升到接近不透明。
- 后方截图必须可点击，不能设置 `pointer-events: none`。
- 后方截图的可见区域必须足够作为点击目标，桌面露出宽度不应少于约 40px；手机不应少于约 28px。
- 不显示底部缩略图列表、胶片条、上下堆叠、常驻左右箭头或图片编号。
- 生产页面不显示“点击提示气泡”或基线说明，除非后续另行确认需要帮助文本。

## 响应式规则

桌面 `1440px` 视口以原型为准：主图居中，后方两张图分别向左上和右上错位，旋转 `-5deg` 与 `5deg`，底部不低于主图。

手机 `390px` 视口需要收敛：

- 主图宽度按当前移动端比例缩小。
- 后方图片旋转角度约 `4deg`。
- 横向偏移减少，确保不挤压 hero 文案和主要操作按钮。
- 后方图片仍需保持可点击露出区域，不得退回到几乎不可见的浅影。

## 状态保持

每个功能场景可以独立保存当前主图。用户在场景 A 内点击第 2 张图成为主图后，切到场景 B 再切回场景 A，应优先保持场景 A 的第 2 张图为主图。语言切换不得重置当前功能场景或当前主图。

自动轮播切到新场景时，显示该场景已保存的主图；若该场景未被用户操作过，则显示默认主图。

## 测试与验收

Vitest 需要覆盖：

- 场景点点可以切换到对应功能场景。
- 自动轮播只改变功能场景，不改变同场景主图。
- 点击后方图片只改变当前场景主图，不改变功能场景。
- 单图场景不显示后方牌堆。
- 后方图片是可访问控件，不是 `pointer-events: none` 的装饰。

Playwright 需要覆盖桌面和手机：

- 1440px 与 390px 下，牌堆后方图片底部不低于主图底部。
- 点击后方图片后，它升到主图。
- 点击场景点点后切换到对应功能场景。
- 拖动或滑动只切功能场景。
- 页面不出现底部缩略图、胶片条、常驻左右箭头或 `01/02/03/04` 编号。

视觉截图需要保存到对应 feature 的 `visual-review/`，并包含桌面和手机的牌堆状态。截图验收时，以本文原型为准比较主图、后方图、底部基线关系、透明度和点击露出面积。

## 不在本轮范围

- 不重新设计首页 hero 左侧文案、下载按钮、在线查询区或页脚。
- 不改变 App 功能事实、三语文案策略、真实 logo 或联系邮箱。
- 不新增、修改或移除后端 HTTP API。
- 不新增底部缩略图、胶片条或常驻箭头作为替代控制。
